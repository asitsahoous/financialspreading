## Progress log

**2026-07-02 — Phase 0 slice shipped: Gate 1–5 sign-off is now real.** Implemented per [ROADMAP.md](ROADMAP.md) Phase 0 (narrowed scope — see below). Concretely:
- Provisioned a dedicated Supabase Postgres project (`financial-spreading-board`, us-east-1) for the backend checkpointer. **DB password still needs to be pasted into `backend/.env`** (Supabase doesn't expose it via API) — until then the backend correctly falls back to local SQLite.
- **Found and fixed 3 previously-unknown backend bugs that meant the backend had never actually run end-to-end before:** (1) `SqliteSaver.from_conn_string()`/`PostgresSaver.from_conn_string()` are `@contextmanager` generators — assigning them directly (as the original code did) yields the wrapper object, not a usable saver, crashing the app on startup; (2) `backend/api/router.py` awaited `graph.ainvoke()` (async) while using sync `SqliteSaver`/`PostgresSaver`, which only support sync methods — every case-creation and gate-sign call raised `NotImplementedError`; (3) `decision.get("reason", default)` in `backend/graph/gates.py` doesn't fall back when `reason` is explicitly `None` (the common case — signing a gate without a reason) because the key *exists*, just with a `None` value — this crashed the `/audit` endpoint via a Pydantic validation error. All three are fixed.
- Wired `app/src/acos/backendCase.ts` (new) + `app/src/acos/FinancialSpreadingACOS.tsx`'s `CaseWorkspaceView` so Gate 1–5 "sign" actions for the Walmart and Northern Retail demo cases call the real backend (`POST /cases/{id}/gates/{gate_id}`) and persist across reload/restart — verified via a real server-restart test and a client-state-wipe test (cleared local audit log, kept only the backend case pointer, reloaded — Gate 2 still recognized as signed).
- **Scope was narrowed from the original plan** for one concrete reason discovered mid-implementation: the frontend's fixture intake-document names (e.g. Walmart's `"WLBOR"`, `"WLBBSHEET"`; Northern Retail's `"FY2024 Annual Report"`) don't match the backend's fixed 9-item SOP §4.2 manifest names (`"10-K Annual Filing"`, etc.) at all — different names, different counts. Wiring the **document intake checklist** itself to the backend's `receive-document` endpoint was deferred rather than forcing a mismatched integration; **document intake, mapping/exceptions, overrides, Gate 5 decline/table, and portfolio widgets remain on local state for now** — still accurate as P0/P1 items below. Reconciling the document schema (one canonical manifest, used by both frontend copy and backend) is necessary follow-up before those can be wired the same way.
- All 24 existing Playwright e2e tests (`npm run test:e2e`) still pass against the wired build — no regressions (one run hit a pre-existing flaky timing issue on Journey 4 unrelated to this change; passed in isolation and on rerun).

**2026-07-02 (same day, follow-on) — Trust Inspector "Accept mapping" / "Apply correction" now real too, for the one field that's safe to wire.** Extended `backendCase.ts` with `overrideBackendField()` and a `BACKEND_WIRED_MAPPING_FIELDS` allowlist (currently just `"Total Assets"` — the one exception field seeded identically, same name and same `$100K` value, on both the frontend fixture and the backend's `document_intel.py`). `CaseWorkspaceView`'s central `appendAudit` dispatcher now calls `POST /cases/{id}/overrides` for `mapping-accept`/`mapping-override` actions on that field, verified via a real network call + a direct check of the resulting `audit_trail` entry in backend state. Every other mapping field stays local-only until the broader document/mapping fixture-vs-backend schema reconciliation happens (same root cause as the deferred document-intake wiring above).

**2026-07-02 (same day, follow-on 2) — Document intake is now real, closing the gap flagged above.** Renamed `CASES.walmart.intakeDocs` (was 4 ad-hoc short-code names — `"WLBOR"`, `"WLBBSHEET"`, etc.) and `CASES["northern-retail"].intakeDocs`'s one mismatched entry (`"FY2024 Annual Report"` → `"10-K Annual Filing"`) to match `backend/graph/nodes/intake.py`'s SOP §4.2 manifest exactly. Added `POST /cases/{id}/documents/{doc_name}/receive` wiring end-to-end: `app/src/api/client.ts` gained `receiveDocument()`, `backendCase.ts` gained `receiveBackendDocument()` + `fetchBackendDocuments()`, and `markDocReceived()` in `CaseWorkspaceView` now calls the real backend and seeds `intakeDocOverrides` from backend state on load (mirroring the gate-sync pattern). Verified with the same two-part test used for gates: a live network call showing the backend's missing-doc count drop from 2/9 to 3/9, and a reload with local session state wiped (kept only the backend case pointer) showing the received document still marked "Extracted" with the correct backend-recorded actor/timestamp. Rebuilt and reran the full Playwright suite — one test (`Journey 5`) asserted the old `"04/04 Documents Uploaded"` header, which was a correct catch of the intentional 4→9 document count change; updated to `"09/09"`. 24/24 pass.

**2026-07-02 (same day, follow-on 3) — Gate 5 Decline/Table wired, and a serious off-by-one bug found and fixed along the way.** Extended the backend's `GateDecision`/`GateDecisionRequest` status enum with a fourth value, `"tabled"` (previously only `approved`/`override`/`rejected`), and wired the two Gate 5 committee buttons through `signBackendGate(..., status)`.

While verifying this carefully (pacing each gate click and checking the network response, rather than firing them in quick succession), found that **every gate signature since the very first Phase 0 commit had been recorded one gate behind** — e.g. clicking "Sign Gate 2" was actually resuming and recording Gate 1, "Sign Gate 3" recorded Gate 2, and so on, with the true last gate (Gate 5) never actually reached. Root cause: `backend/graph/gates.py`'s gate nodes each pause on their own `interrupt()` call, and the graph is additionally paused *before* every gate node via `interrupt_before=[...]` in `graph.py`. The Walmart happy-path UI starts directly at the "Review" (Gate 2) screen and never calls Gate 1's sign endpoint — but the backend graph still needs an explicit resume to advance past Gate 1's interrupt before *any* later gate's resume value will land on the right node. Every subsequent "sign" call was consumed by the still-pending *previous* gate instead. Confirmed via an isolated Python reproduction (gates signed in strict order 1→5 worked perfectly) versus the actual browser flow (which skips Gate 1) before identifying the fix.

Fix: `ensureBackendCase()` in `backendCase.ts` now auto-signs Gate 1 on the backend immediately after creating a Walmart-role case (whose seed data has all 9 documents already received, matching its "intake already complete" narrative) — before returning the case ID to the caller, so every later gate call lands on the gate the UI actually intended. Northern Retail is unaffected: its seed is intentionally partial, and Gate 1 is signed for real through the Intake UI once documents are received, so its interrupt cursor was never out of sync.

Verified with the same paced-click methodology across all 5 gates end-to-end (`gate1: approved, gate2: approved, gate3: approved, gate4: approved, gate5: tabled` — each correctly attributed). Also confirms every prior "Gate N signed" verification claim in this log up to this point was technically checking gate N-1's record — the *persistence and reload-safety* findings were still valid (the mechanism works), but the specific gate attribution was wrong until this fix. Rebuilt and reran the full Playwright suite: 24/24 pass.

**Where this leaves things:** the entire Command Center → Cases → 5-gate workflow (case creation, document intake, all 5 gate outcomes including Decline/Table, the "Total Assets" mapping correction, audit trail) is now backend-real and *correctly attributed* for Walmart and Northern Retail. What's still local/fixture-only: mapping fields other than "Total Assets", portfolio widgets for the other borrowers (AutoWest/Costco/Target — still routed to Walmart per the documented shortcut), and the Financial Spread tab (separate subsystem, out of scope for this phase per docs/ROADMAP.md).

**2026-07-03 — Mapping-field reconciliation extended beyond "Total Assets".** Compared `MAPPING_CORE_ROWS` (Walmart's 6 "real" mapping rows) against the backend's `document_intel.py` `FIELD_TEMPLATES`/`mapping.py` `COA_MAP`: 5 of 6 already matched by name (`Total Assets`, `Long-term Debt`, `Shareholders Equity`, `Revenue`, `Cash & Equivalents`); renamed the 6th (`"Accounts Receivable"` → `"Receivables, net"`) to match exactly. All 6 are now in `BACKEND_WIRED_MAPPING_FIELDS` and verified end-to-end (Revenue accept → real `POST /overrides` → confirmed in backend audit trail). The other ~134 rows in the Walmart grid (`MAPPING_SYNTHETIC_FIELDS`) are auto-generated filler used only to pad the demo out to "138/140 fields" — they have no backend counterpart at all (the backend only models 24 real fields total), and giving them one would mean inventing ~110 more backend fields with no unique narrative. Left local-only as a deliberate scope cut, not an oversight. 24/24 Playwright tests pass.

**2026-07-02 (same day, follow-on 4) — Concurrent-request risk closed.** The off-by-one bug above was proven to be purely sequential/logical, not a demonstrated concurrency race — but the underlying read-modify-write pattern in `sign_gate`, `receive_document`, and `override_field` (`get_state` → mutate → `invoke`/`update_state`) was still genuinely unsafe against two truly-overlapping requests for the same `case_id`. Added a per-`case_id` `asyncio.Lock` (`_case_locks`/`_lock_for` in `router.py`) around each of those three endpoints' critical sections; `create_case` doesn't need one since every case gets a fresh server-generated UUID. Verified with a real concurrency test — fired all 5 gate-sign requests for one case via `asyncio.gather` (true simultaneous overlap, not paced) and confirmed all 5 landed correctly attributed. 24/24 Playwright tests still pass.

## Not a defect — confirmed correct behavior

The audit also checked several places that *look* like bugs but are intentional and correct, so they're explicitly out of scope for remediation:

- Gate sign buttons disabling after signing (audit immutability, by design — becomes DB-enforced per P2-1)
- Mapping page Next/Previous pagination disabling at bounds (real pagination logic)
- "Save comment" disabled on empty input (real validation)
- Toasts auto-clearing after ~2.8s (correct — toasts are meant to be ephemeral, unlike the P1-6 items whose *only* effect is the toast)
